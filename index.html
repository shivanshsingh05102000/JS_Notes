Introduction to JavaScript

Created to make web pages alive. Scripts run as plain text — no compilation needed. Originally "LiveScript", renamed to ride Java's popularity. Fully independent today; its spec is called ECMAScript.

JS Engines by Browser

EngineBrowser
V8Chrome, Opera, Edge
SpiderMonkeyFirefox
ChakraIE
JavaScriptCoreSafari

Engine Pipeline

Parse → Compile to machine code → Run (with optimisations at each step)

What JS Can Do (Browser)

  • Add/change HTML content and CSS styles
  • React to user actions (clicks, keys, mouse)
  • Make network requests (AJAX, COMET)
  • Store data client-side (localStorage)

What JS Cannot Do (Browser)

Read/write files, access OS, use camera/mic without permission, access other-domain tabs (Same Origin Policy)

Old comment trick — code wrapped in <!-- ... //--> inside <script> was used to hide JS from ancient browsers. Seeing this = very old code.

External Scripts

<script src="/path/to/script.js"></script>   // absolute path
<script src="script.js"></script>             // relative path
<script src="https://example.com/script.js"></script>  // full URL

// Multiple scripts = multiple tags:
<script src="/js/script1.js"></script>
<script src="/js/script2.js"></script>

Why External Files?

  • Browser downloads once and caches it
  • Other pages reuse the cache → less traffic, faster load

src + Inline Code Don't Mix

If src is set, any inline code inside the tag is ignored.

// WRONG — alert(1) is ignored
<script src="file.js">
  alert(1);
</script>

// CORRECT — split into two tags
<script src="file.js"></script>
<script>alert(1);</script>

Code Structure

Instructions are called statements, separated by semicolons. JS has automatic semicolon insertion (ASI) — but it's unreliable.

Never rely on ASI. Always write semicolons manually.
// dangerous — JS merges these into one broken statement
alert("Hello")
[1, 2].forEach(alert);  // reads as: alert("Hello")[1,2].forEach(...)

Comments

// single line comment

/* multi-line
   comment */

/* alert('disabled'); */  // temporarily disable code
Nested comments /* /* */ */ throw an error.

"use strict"

ES5 (2009) introduced fixes that are off by default to avoid breaking legacy code. "use strict" opts into these fixes.

"use strict";
// entire script runs in strict mode
Must be at the very top — placing it anywhere else is ignored. Cannot be undone.
Classes and modules enable strict mode automatically — once you use them, "use strict" is redundant.

Comparisons

Strict vs Loose Equality

0 == false    // true  — type coercion
0 === false   // false — strict, no conversion
"" == false   // true  — loose
"" === false  // false — strict
Always use ===. Never use == unless you specifically need type coercion.

null & undefined Traps

null == undefined   // true  (special pair)
null === undefined  // false (different types)

null >= 0    // true  — null converts to 0
null == 0    // false — == uses different rules, null only == undefined
undefined    // always false in comparisons (converts to NaN)
Never use > < >= <= with variables that could be null/undefined. Check first.

Conditional Branching: if and ?

Runs code only if the condition is true. Condition is converted to boolean automatically.

if / else if / else

if (year < 2015) {
  alert("Too early");
} else if (year > 2015) {
  alert("Too late");
} else {
  alert("Exactly!");
}
Always use curly braces {} even for single lines — better readability.

Falsy vs Truthy in conditions

0 "" null undefined NaN everything else → truthy

Ternary Operator ?

let result = condition ? value1 : value2;

let accessAllowed = (age > 18) ? true : false;
// Even simpler — comparison already returns boolean:
let accessAllowed = age > 18;

// Chaining:
let message = (age < 3) ? 'Hi, baby!' :
  (age < 18) ? 'Hello!' :
  (age < 100) ? 'Greetings!' :
  'What an unusual age!';

? vs if — When to Use What

Use ?Use if
Assigning a value based on conditionExecuting different blocks of code
Short, simple conditionsComplex logic, multiple statements
Don't use ? as a replacement for if to execute code — it hurts readability. Use if for actions.

Logical Operators

|| && ! work on any type, return original values, not just true/false.

|| OR — first truthy
Stops at first truthy. Returns that original value. If all falsy → returns last.
&& AND — first falsy
Stops at first falsy. Returns that original value. If all truthy → returns last.
1 || 0          // 1  (first truthy)
null || 1       // 1  (skipped null)
null || 0 || 1  // 1  (first truthy found)
undefined || null || 0  // 0  (all falsy, returns last)

1 && 0          // 0  (first falsy)
1 && 2 && 3     // 3  (all truthy, returns last)
null && 5       // null (stops immediately)

!! Double NOT Trick

!!"hello"  // true  — converts any value to pure boolean
!!null     // false
!!0        // false
// same as Boolean(value)

Precedence

! (highest) → && → || (lowest)

Nullish Coalescing ??

Returns the first defined value — only null and undefined trigger fallback.

?? vs || — The Critical Difference

OperatorFallback triggers onKeeps
||Any falsy: 0, "", false, null, undefinedOnly truthy
??Only null and undefined0, "", false
 
let height = 0;
height || 100   // 100 — WRONG, loses the 0
height ?? 100   // 0   — CORRECT, respects user's value

// Real example — volume control
let volume = 0;
volume || 50    // 50  — bug, overwrites setting
volume ?? 50    // 0   — correct
Mixing ?? with && or || without parentheses throws a SyntaxError.

The switch Statement

Replaces multiple if checks with a more descriptive way to compare a value against multiple variants.

Syntax

switch(x) {
  case 'value1':   // if (x === 'value1')
    ...
    [break]
  case 'value2':
    ...
    [break]
  default:
    ...
    [break]
}
  • Uses strict equality (===) — type and value must both match
  • Execution starts from matched case and runs until nearest break or end of switch
  • If no case matches, default block runs (if it exists)

Fall-through (no break)

Without break, execution continues into the next case automatically — no check is done.

let a = 4;
switch (a) {
  case 3: alert('Too small');
  case 4: alert('Exactly!');   // starts here
  case 5: alert('Too big');    // also runs!
  default: alert("Unknown");   // also runs!
}

Grouping Cases

case 3:
case 5:
  alert('Wrong!');
  break;
Type matters: prompt() always returns a string. case 3 (number) will never match "3" (string) — mismatched types create silent dead code.

Coding Style

Curly Braces — Egyptian Style

Opening brace on same line, space before bracket.

if (condition) {   // ✅ correct
  // code
}

if (condition)     // ❌ never split without braces
  doSomething();

One-liner acceptable only if very short: if (n < 0) return null;

Line Length

Max 80 or 120 chars (team decides). Split long conditions:

if (
  id === 123 &&
  moonPhase === 'Waning Gibbous'
) {
  letTheSorceryBegin();
}

Indents

  • Horizontal — 2 or 4 spaces (spaces preferred over tabs)
  • Vertical — empty lines between logical blocks. No more than 9 lines without a break.

Nesting — Keep It Shallow

Use early returns and continue to avoid extra levels:

// ❌ nested
if (cond) { doSomething(); }

// ✅ flat
if (!cond) continue;
doSomething();

Style Guides & Linters

  • Popular guides: Google, Airbnb, StandardJS
  • Linters autocheck style AND catch bugs — use ESLint
npm install -g eslint
// .eslintrc:
{
  "extends": "eslint:recommended",
  "env": { "browser": true, "es6": true },
  "rules": { "indent": 2 }

Comments

Bad Comments — explaining WHAT the code does

If you need to explain what, rewrite the code instead. A good function name IS the comment.

// ❌ bad — the comment just restates the code
// loops through items and checks if prime
for (let i = 2; i < n; i++) { ... }

// ✅ no comment needed — function name explains it
if (!isPrime(i)) continue;

Good Comments — comment the WHY, not the WHAT

  • Architecture — high-level overview of how components interact
  • Function docs (JSDoc) — parameters, return value, usage
  • Why this solution — explain why you chose this approach, especially if a more obvious solution was tried and rejected
/**
 * Returns x raised to the n-th power.
 * @param {number} x The number to raise.
 * @param {number} n The power, must be natural.
 * @return {number} x raised to the n-th power.
 */
function pow(x, n) { ... }
Comment thisDon't comment this
Overall architectureWhat the code does
Function usage/paramsHow a loop works
Non-obvious decisionsThings a good function name already says
Absence of comments is also a skill — clean code that doesn't need them is the goal.

Automated Testing with Mocha

Manual re-runs miss things. Fix f(2), forget to recheck f(1). Automated tests catch regressions automatically.

BDD — Behavior Driven Development

Spec = tests + documentation + examples in one. Write the spec first, then implement.

  • Write spec for basic functionality
  • Write initial implementation
  • Run tests — fix until they pass
  • Add more test cases → tests fail → update implementation → tests pass
  • Repeat until done

Spec Structure

describe("pow", function() {        // group — what are we testing?
  it("raises to n-th power", function() {  // one use case
    assert.equal(pow(2, 3), 8);            // check expected result
  });
});
One it = one thing. If you have two independent checks, split into two it blocks — when an assert fails, the it block stops and you lose info about what else failed.

Hooks

before(() => ...)       // runs once before ALL tests
after(() => ...)        // runs once after ALL tests
beforeEach(() => ...)   // runs before EACH it
afterEach(() => ...)    // runs after EACH it

Common Chai Assertions

AssertionChecks
assert.equal(a, b)a == b (loose)
assert.strictEqual(a, b)a === b (strict)
assert.isTrue(v)v === true
assert.isFalse(v)v === false
assert.isNaN(v)v is NaN
assert.notEqual(a, b)inverse of equal
Libraries: Mocha (test runner), Chai (assertions), Sinon (spies, mocks, stubs). Without tests — either ship bugs or become too afraid to touch old code.

Polyfills & Transpilers

JS evolves fast. New features don't work on older engines. Two tools fix this.

Transpiler

Converts modern syntax to older equivalent syntax. Most popular: Babel. Runs automatically via webpack.

// your code
height = height ?? 100;

// after transpiler (Babel):
height = (height !== null && height !== undefined) ? height : 100;

Polyfill

Adds missing built-in functions that older engines don't have. Popular library: core-js.

if (!Math.trunc) {
  Math.trunc = function(number) {
    return number < 0 ? Math.ceil(number) : Math.floor(number);
  };
}

Which One to Use?

ProblemTool
New syntax/operators (??, =>)Transpiler
Missing functions (Math.trunc, Array.from)Polyfill
BothBoth
Write modern code, let tooling handle backwards compatibility. Don't dumb down your code manually.

Objects

Used to store keyed collections of data and complex entities. Objects penetrate almost every aspect of JavaScript.

Creating an Object

let user = new Object(); // object constructor syntax
let user = {};           // object literal syntax (preferred)

let user = { name: "John", age: 30 };  // with properties inline
A trailing comma after the last property is allowed and encouraged — makes editing easier.

Accessing Properties

user.name           // dot notation — simple keys only
user["likes birds"] // square brackets — any string, including multiword
user[key]           // square brackets accept variables and expressions
                    // dot notation CANNOT do this

Adding, Modifying, Deleting

user.isAdmin = true;   // add or update
delete user.age;       // delete a property
// Accessing a non-existent property returns undefined — no error thrown

Computed Properties

let fruit = prompt("Which fruit?", "apple");
let bag = {
  [fruit]: 5,              // key taken from variable
  [fruit + 'Computers']: 5 // expressions also work
};

Property Value Shorthand

// Instead of: { name: name, age: age }
function makeUser(name, age) {
  return { name, age };  // shorthand — same result
}

Checking if a Property Exists

user.noSuchProp === undefined  // works in most cases
"key" in object                // safer — the "in" operator

// Why "in" is safer:
let obj = { test: undefined };
obj.test              // undefined — looks like missing!
"test" in obj         // true — correctly detects it exists

The for..in Loop

for (let key in user) {
  alert(key);        // property name
  alert(user[key]);  // property value
}
// Integer properties are sorted numerically
// Non-integer properties appear in creation order

Property Name Limitations

  • No restrictions — reserved words like for, let, return are all valid as keys
  • Non-string keys auto-convert to strings — number 0 becomes string "0"
  • Special case: __proto__ cannot be set to a non-object value — silently ignores the assignment

Object References & Copying

Primitives (string, number, bool) are copied by value — two independent copies. Objects are copied by reference — both variables point to the same object.

Reference Copying

let user = { name: "John" };
let admin = user; // same object, not a new one
admin.name = "Pete";
alert(user.name); // "Pete" — both affected

Comparison by Reference

let a = {};
let b = a;   // same reference
a == b  // true

let c = {};
let d = {};  // two separate empty objects
c == d  // false — different references

const Objects

const prevents reassigning the variable, not changing its properties.

const user = { name: "John" };
user.name = "Pete"; // ✅ works fine — changing a property
user = {};          // ❌ error — reassigning the variable

Cloning (Shallow)

// Method 1 — loop
for (let key in user) { clone[key] = user[key]; }

// Method 2 — Object.assign
let clone = Object.assign({}, user);

// Method 3 — spread (preferred)
let clone = { ...user };
Shallow clone fails for nested objects — they still share the same reference inside.

Deep Cloning (Nested Objects)

let clone = structuredClone(user); // fully independent copy
// Works with: objects, arrays, primitives, circular references
// ❌ Fails with: function properties

// Alternative: _.cloneDeep(obj) from lodash
structuredClone handles circular references — JSON.parse(JSON.stringify(obj)) does not. Prefer structuredClone.

Garbage Collection

JS automatically manages memory — you don't do it manually. The garbage collector removes objects that are no longer reachable.

Reachability — The Core Concept

A value is reachable if it can be accessed somehow → stays in memory. Always-reachable roots:

  • Currently running function + its variables
  • Nested call chain variables
  • Global variables

Anything reachable from a root (via references) also stays in memory.

Examples

let user = { name: "John" }; // reachable
user = null; // object now unreachable → gets deleted

// Two references:
let user = { name: "John" };
let admin = user;
user = null; // object still alive — admin holds it
admin = null; // NOW deleted
Outgoing references don't matter — only incoming ones keep an object alive. A whole group of interlinked objects is deleted if nothing from root points to them.

Mark-and-Sweep Algorithm

  • Mark all roots
  • Follow & mark all their references
  • Repeat until all reachable objects are marked
  • Delete everything unmarked

Engine Optimizations

OptimizationWhat it does
GenerationalTracks "new" objects separately — most objects are short-lived
IncrementalSplits cleanup into small chunks, avoids big pauses
Idle-timeRuns only when CPU is free

Object Methods & this

A method is just a function stored as a property of an object.

user.sayHi = function() { alert("Hello"); }

// shorthand (preferred):
let user = {
  sayHi() {
    alert("Hello");
  }
}

What is this?

this inside a method refers to the object before the dot at call time, not definition time.

let user  = { name: "John",  sayHi() { alert(this.name); } };
let admin = { name: "Admin", sayHi: user.sayHi };

user.sayHi();  // "John"  — this = user
admin.sayHi(); // "Admin" — this = admin
Never reference the object by variable name inside a method. If that variable gets reassigned or nulled, it breaks silently. 
// ❌ WRONG — fragile
sayHi() { alert(user.name); }

// ✅ CORRECT — always points to caller
sayHi() { alert(this.name); }

Calling a Function With No Object Context

  • Strict mode → this is undefined → accessing this.anything throws
  • Non-strict mode → this becomes the global object (window in browser)
Both are bugs. A function using this expects to be called on an object.

Arrow Functions Have No this

Arrow functions inherit this from the outer lexical scope — whatever this was where the arrow function was written.

let user = {
  firstName: "Ilya",
  sayHi() {
    let arrow = () => alert(this.firstName); // borrows this from sayHi
    arrow();
  }
};
user.sayHi(); // "Ilya"
Use arrow functions when you don't want a new this context — e.g. callbacks inside methods.

Constructor Functions & new

Object literals {} create one object. Constructors create many similar objects with reusable code.

The Two Conventions

  • Name starts with a capital letter
  • Always called with new
These are agreements, not enforced rules. Break them and things still work — just becomes unreadable and error-prone.

What new Does Internally

function User(name) {
  // 1. this = {}  ← empty object created implicitly
  this.name = name;
  this.isAdmin = false;
  // 2. return this  ← returned implicitly
}

let user = new User("Jack");
// user = { name: "Jack", isAdmin: false }
  • Creates a new empty object, assigns it to this
  • Runs the function body — you populate this
  • Returns this automatically

Methods Inside Constructors

function User(name) {
  this.name = name;
  this.sayHi = function() {
    alert("My name is: " + this.name);
  };
}

Return Behavior (Edge Case)

Constructors normally have no return. But if they do:

return valueResult
An objectThat object is returned — this is discarded
A primitive (or nothing)Ignored — this is returned as normal
 
function BigUser() {
  this.name = "John";
  return { name: "Godzilla" }; // this is thrown away
}
new BigUser().name // "Godzilla"

new.target (Advanced)

function User(name) {
  if (!new.target) return new User(name); // auto-fix missing new
  this.name = name;
}
// Called normally → new.target is undefined
// Called with new → new.target is the function itself
Core mental model: a constructor is a blueprint. new is the machine that stamps out objects from it. Every call to new creates a fresh isolated object populated by whatever you assign to this.

Symbol Type

Only two types may serve as object property keys: string or symbol. All other types auto-convert to string (e.g. obj[1] is the same as obj["1"]).

What is a Symbol?

A guaranteed unique identifier. Even symbols with the same description are different values.

let id = Symbol("id");          // symbol with description "id"

let id1 = Symbol("id");
let id2 = Symbol("id");
id1 == id2  // false — always unique

Symbols Don't Auto-Convert to String

let id = Symbol("id");
alert(id);           // TypeError — can't auto-convert
alert(id.toString()); // "Symbol(id)" — explicit works
alert(id.description); // "id" — just the description

Hidden Properties

Symbols let you add properties to objects that third-party code won't accidentally access or overwrite.

let user = { name: "John" }; // belongs to another codebase
let id = Symbol("id");
user[id] = 1;
alert(user[id]); // 1

// If another script also uses Symbol("id"), there's no conflict:
let theirId = Symbol("id");
user[theirId] = "their value"; // totally independent
Symbols are skipped by for...in and Object.keys() — they are invisible to standard iteration. But Object.assign() does copy symbols.

Symbols in Object Literals

let id = Symbol("id");
let user = {
  name: "John",
  [id]: 123    // ← must use square brackets, not "id": 123
};

Global Symbol Registry

Use when you need the same symbol across different parts of the codebase.

// Create or retrieve from global registry
let id = Symbol.for("id");

// Retrieve again — same symbol guaranteed
let idAgain = Symbol.for("id");
id === idAgain  // true

// Get name by symbol
Symbol.keyFor(id)      // "id"
Symbol.keyFor(Symbol("local"))  // undefined — not global

System Symbols

JS uses built-in well-known symbols internally to fine-tune object behavior:

Symbol.toPrimitive
Controls object → primitive conversion
Symbol.iterator
Makes objects iterable (for...of)
Symbol.hasInstance
Customizes instanceof behavior
Symbol.isConcatSpreadable
Controls Array.prototype.concat behavior

Object → Primitive Conversion

Objects cannot do math with each other. When you use an object in a numeric or string context, JS forcibly converts it to a primitive. The result is always a primitive — you cannot get an object back from obj1 + obj2.

Boolean conversion doesn't exist here — all objects are always true. Full stop.

Three Hints — How JS Decides

HintWhen it triggers
"string"alert(obj), using obj as a property key
"number"Math operations, unary +, comparisons > <
"default"Binary +, == with a primitive — operator is unsure of type
All built-in objects treat "default" the same as "number". Effectively it's just string vs number.

Conversion Method Priority Order

  • obj[Symbol.toPrimitive](hint) — if it exists, used for all hints. Done.
  • If hint is "string" → try toString() first, then valueOf()
  • If hint is "number" or "default" → try valueOf() first, then toString()

Symbol.toPrimitive — The Clean Modern Way

let user = {
  name: "John",
  money: 1000,

  [Symbol.toPrimitive](hint) {
    return hint == "string" ? `{name: "${this.name}"}` : this.money;
  }
};

alert(user);       // {name: "John"} — string hint
alert(+user);      // 1000           — number hint
alert(user + 500)  // 1500           — default hint → number

toString / valueOf — The Old Way

let user = {
  name: "John",
  money: 1000,
  toString() { return `{name: "${this.name}"}`; }, // string hint
  valueOf()  { return this.money; }                 // number/default hint
};

// Same result as Symbol.toPrimitive above, just split across two methods

Default Behavior (No Custom Methods)

let user = { name: "John" };
alert(user);      // "[object Object]" — toString default
user.valueOf()    // returns the object itself → ignored (not a primitive)

If you only define toString and nothing else — it handles all conversions as fallback. Useful for simple cases.

Return Type Rules

  • Methods must return a primitive — any primitive, not necessarily the hinted type
  • If toString/valueOf returns an object → silently ignored, moves to next method
  • If Symbol.toPrimitive returns an object → throws an error. Stricter.

Further Conversions — Two Stage Process

let obj = { toString() { return "2"; } };

alert(obj * 2);  // 4   → stage 1: obj → "2", stage 2: "2" * 2 → 4
alert(obj + 2);  // "22" → stage 1: obj → "2", stage 2: "2" + 2 → concat!
Binary + with a string result will concatenate, not add. This is a common gotcha in two-stage conversion.
